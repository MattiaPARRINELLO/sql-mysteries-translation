Le mystère du meutre SQL

Pouvez-vous découvrir le coupable ?

Il y a eu un meutre dans la ville SQL! Le mystère du meurtre SQL est designé pour être à la fois une leçon en autonomie pour apprendre les conceptes et commandes du SQL et pour être un jeu amusant pour les utilisateurs plus avancés de SQL et ainsi résoudre un crime.

Guide pour les débutants en SQL

Si tu es à l'aise avec SQL, vous pouvez sautez ces explications et mettre tes compétences à l'épreuve! Ci-dessous, nous introduisons quelques concepts de base du SQL, et juste assez de détails pour résoudre le meurtre. Si tu veux une introduction plus complète au SQL, essaye Select

Un crime a eu lieu et le détective a besoin de votre aide. Le détective t'as donné le rapport de la scène de crime, mais tu l'as perdu. Tu te rappel vaguement que le crime était un meurtre qui a eu lieu à un moment donné le 15 janvier 2018 et qu'il a eu lieu à SQL City. Commence par récupérer le rapport de la scène de crime correspondant dans la base de données du département de police.

Tous les indices de ce mystère sont enfouis dans une énorme base de données, et tu dois utiliser SQL pour naviguer à travers ce vaste réseau d'informations. Ton premier pas pour résoudre le mystère est de récupérer le rapport de la scène de crime correspondant dans la base de données du département de police. Ci-dessous, nous expliquerons de manière générale les commandes que tu dois connaître ; quand tu seras prêt, tu pourras commencer à adapter les exemples pour créer tes propres commandes SQL à la recherche d'indices - tu peux exécuter n'importe quel SQL dans n'importe quelle boîte de code, peu importe ce qu'il y avait dans la boîte lorsque tu as commencé.

Quelques définitions

Qu'est ce que le SQL

SQL, qui signifie Structured Query Language, est une façon d'interagir avec des bases de données relationnelles et des tables de manière à nous permettre, en tant qu'humains, d'obtenir des informations spécifiques et significatives.

Attends, qu'est-ce qu'une base de données relationnelle ?

Il n'y a pas de définition unique pour le mot base de données. En général, les bases de données sont des systèmes de gestion de l'information. Les bases de données peuvent avoir des degrés de structure différents imposés sur les données. Lorsque les données sont plus structurées, cela peut aider les personnes et les ordinateurs à travailler plus efficacement avec les données.

Les bases de données relationnelles sont probablement le type de base de données le plus connu. Au cœur des bases de données relationnelles se trouvent des tables, qui ressemblent beaucoup à des feuilles de calcul. Chaque colonne de la table a un nom et un type de données (texte, nombre, etc.), et chaque ligne de la table est une instance spécifique de ce dont parle la table. La partie "relationnelle" vient avec des règles spécifiques sur la façon de connecter les données entre différentes tables.

Qu'est-ce qu'un ERD ?

ERD, qui signifie Entity Relationship Diagram, est une représentation visuelle des relations entre toutes les tables pertinentes d'une base de données. Vous pouvez trouver l'ERD de notre base de données SQL Murder Mystery ci-dessous. Le diagramme montre que chaque table a un nom (en haut de la boîte, en gras), une liste de noms de colonnes (à gauche) et leurs types de données correspondants (à droite, en majuscules). Il y a aussi des icônes de clé dorée, des icônes de flèche bleue et des flèches grises sur l'ERD. Une clé dorée indique que la colonne est la ​clé primaire​ de la table correspondante, et une flèche bleue indique que la colonne est la ​clé étrangère​ de la table correspondante.

Clé primaire:
un identifiant unique pour chaque ligne dans une table.
Clé étrangère:
utilisée pour référencer des données d'une table à une autre table.
Si deux tables sont liées, les colonnes correspondantes, c'est-à-dire les identifiants communs des deux tables, sont connectées par une flèche grise dans le diagramme.

Qu'est-ce qu'une requête ?

Si tu regardes les données de cette base de données, tu verras que les tables sont énormes ! Il y a tellement de points de données ; il n'est tout simplement pas possible de parcourir les tables ligne par ligne pour trouver les informations dont nous avons besoin. Que devons-nous faire ?

C'est là que les requêtes entrent en jeu. Les requêtes sont des instructions que nous construisons pour obtenir des données à partir de la base de données. Les requêtes se lisent comme de l'anglais courant (pour la plupart). Essayons quelques requêtes sur notre base de données. Pour chacune des boîtes ci-dessous, clique sur "exécuter" pour exécuter la requête dans la boîte. Tu peux modifier les requêtes directement ici sur la page pour explorer davantage. (Note que les commandes SQL ne sont pas sensibles à la casse, mais il est courant de les capitaliser pour plus de lisibilité. Tu peux également utiliser des nouvelles lignes et des espaces blancs comme tu le souhaites pour formater la commande de manière lisible. La plupart des systèmes de base de données exigent que tu termines une requête par un point-virgule (';'), bien que le système pour les exécuter sur cette page web soit plus tolérant.)

Quels éléments contient une requête SQL ?

Une requête SQL peut contenir :

  • Mots-clés SQL (comme ​SELECT​ et ​FROM​ ci-dessus)
  • Noms de colonnes (comme la colonne ​name​ ci-dessus)
  • Noms de tables (comme la table ​person​ ci-dessus)
  • Caractères génériques (comme ​%)
  • Fonctions
  • Critères de filtrage spécifiques
  • Etc

Mots-clés SQL

Les mots-clés SQL sont utilisés pour spécifier des actions dans vos requêtes. Les mots-clés SQL ne sont pas sensibles à la casse, mais nous vous suggérons d'utiliser des majuscules pour les mots-clés SQL afin de les distinguer facilement du reste de la requête. Voici quelques mots-clés fréquemment utilisés :

SELECT

SELECT permet de récupérer des données pour des colonnes spécifiques de la base de données :

  • * (astérisque) : il est utilisé après SELECT pour récupérer toutes les colonnes de la table ;
  • nom_de_colonne(s) : pour sélectionner des colonnes spécifiques, indiquez les noms des colonnes après SELECT et utilisez des virgules pour les séparer.
FROM

FROM permet de spécifier la ou les tables qui nous intéressent ; pour sélectionner plusieurs tables, indiquez les noms des tables et utilisez des virgules pour les séparer. (Mais jusqu'à ce que vous appreniez le mot-clé JOIN, vous pourriez être surpris de ce qui se passe. Cela viendra plus tard.)

WHERE

La clause WHERE dans une requête est utilisée pour filtrer les résultats selon des critères spécifiques.

Essayons quelques-unes de ces choses.

Le mot-clé ​AND​ est utilisé pour chaîner plusieurs critères de filtrage afin que les résultats filtrés répondent à tous les critères. (Il y a aussi un mot-clé OR, qui renvoie les lignes correspondant à l'un des critères.)

(Omet le commentaire initial /*) Si tu as trouvé la requête qui affiche le seul rapport de scène de crime au lieu de plusieurs pour la même ville et le même type, alors félicitations et ignore le mot "incorrect". Tu sauras si tu l'as trouvé !

Jokers et autres fonctions pour les correspondances partielles

Parfois, tu ne connais qu'une partie des informations dont tu as besoin. SQL peut gérer cela. Les symboles spéciaux qui représentent des caractères inconnus sont appelés "jokers", et SQL en prend en charge deux. Le plus courant est le joker %.

Lorsque tu places un joker % dans une chaîne de requête, le système SQL renverra les résultats qui correspondent exactement au reste de la chaîne, et qui ont n'importe quoi (ou rien) à l'emplacement du joker. Par exemple, 'Ca%a' correspond à Canada et California.

L'autre joker, moins couramment utilisé, est _. Celui-ci signifie "correspondre au reste du texte, tant qu'il y a exactement un caractère à exactement la position du _, peu importe ce que c'est. Ainsi, 'B_b' correspondrait à 'Bob' et 'Bub' mais pas à 'Babe' ou 'Bb'.

Important : Lors de l'utilisation de jokers, tu n'utilises pas le symbole = ; à la place, tu utilises LIKE.

SQL prend également en charge les comparaisons numériques telles que < (inférieur à) et > (supérieur à). Tu peux également utiliser les mots-clés BETWEEN et AND - et tout cela fonctionne avec des mots ainsi qu'avec des nombres.

Nous avons mentionné que les commandes SQL ne sont pas sensibles à la casse, mais les valeurs de requête pour = et LIKE le sont. Parfois, tu ne sais pas comment le texte est stocké dans la base de données. SQL fournit quelques fonctions qui peuvent faciliter cela pour toi. Elles s'appellent UPPER() et LOWER(), et tu peux probablement deviner ce qu'elles font, surtout si tu explores dans la boîte ci-dessous.

Fonctions d'agrégation SQL

Parfois, les questions que tu veux poser ne sont pas aussi simples que de trouver la ligne de données qui correspond à un ensemble de critères. Tu peux vouloir poser des questions plus complexes comme "Qui est la personne la plus âgée ?" ou "Qui est la personne la plus petite ?" Les fonctions d'agrégation peuvent t'aider à répondre à ces questions. En fait, tu as déjà appris une fonction d'agrégation, COUNT.

Quel âge a la personne la plus âgée avec un permis de conduire ? Avec une petite quantité de données, tu pourrais simplement l'estimer visuellement, mais il y a des milliers d'enregistrements dans la table drivers_license. (Essaie COUNT si tu veux savoir combien exactement !) Tu ne peux pas simplement parcourir cette liste pour trouver la réponse.

Voici quelques fonctions d'agrégation utiles que SQL fournit :

MAX
trouve la valeur maximale
MIN
trouve la valeur minimale
SUM
calcule la somme des valeurs de colonne spécifiées
AVG
calcule la moyenne des valeurs de colonne spécifiées
COUNT
compte le nombre de valeurs de colonne spécifiées

Il y a une autre façon de trouver les valeurs minimales et maximales, tout en voyant plus de données. Tu peux contrôler l'ordre de tri des résultats que tu obtiens. C'est vraiment assez intuitif : il suffit d'utiliser ORDER BY suivi d'un nom de colonne. Cela peut être difficile quand il y a beaucoup de données ! (Quand les gens deviennent sérieux à propos du travail avec SQL, ils utilisent de meilleurs outils que ce système basé sur le web.) Par défaut, ORDER BY se fait en ordre "ascendant" (du plus petit au plus grand, ou de A à Z), mais tu peux être spécifique avec ASC pour ascendant, ou tu peux le renverser avec DESC.

À ce stade, tu connais suffisamment SQL pour identifier les deux témoins. Essaie !

Établir des connexions

Joindre des tables

Jusqu'à présent, nous avons posé des questions qui peuvent être répondues en considérant les données d'une seule table. Mais que faire si nous devons poser des questions plus complexes qui nécessitent simultanément des données de deux tables différentes ? C'est là que JOIN intervient.

Les utilisateurs SQL plus expérimentés utilisent quelques types de JOIN différents - vous pouvez entendre parler de INNER, OUTER, LEFT et RIGHT joins. Ici, nous parlerons simplement du type de JOIN le plus courant, le INNER JOIN. Comme il est courant, vous pouvez omettre INNER dans vos commandes SQL.

La façon la plus courante de joindre des tables est d'utiliser les colonnes de clé primaire et de clé étrangère. Référez-vous au diagramme d'entité-relation (ERD) ci-dessus si vous ne vous souvenez pas de ce que sont ces clés, ou pour voir les relations clés entre les tables de notre base de données. Vous pouvez effectuer des jointures sur n'importe quelle colonne, mais les colonnes clés sont optimisées pour des résultats rapides. Il est probablement plus facile de montrer comment les jointures fonctionnent avec notre système SQL interactif que de les écrire.

Parfois, tu veux connecter plus d'une table. SQL te permet de joindre autant de tables que tu le souhaites dans une requête.

Maintenant que tu sais comment joindre des tables, tu devrais être capable de trouver les transcriptions des entretiens pour les deux témoins que tu as identifiés précédemment. Essaie !
